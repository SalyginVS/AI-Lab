- Токены загружаются из env `LLM_GATEWAY_TOKENS` — JSON dict `{token: user_id}`

## Regression: RAG MCP → gateway 401

### Симптом

После деплоя 14B Continue Agent mode + RAG MCP перестал работать. Tool call `search_docs` → 401 Unauthorized.

### Root cause

`~/rag-mcp-server/gateway_embeddings.py` вызывал `POST /v1/embeddings` **без Bearer token**. При optional auth (v0.10.0) это работало. При mandatory auth (v0.11.0) → 401.

### Cascade

1. Gateway стал mandatory auth → rag-mcp embeddings call → 401
2. `gateway_embeddings.py` не содержал auth header logic
3. `rag-mcp.service` не имел `EnvironmentFile=` → env `LLM_GATEWAY_TOKENS` не виден процессу

### Fix (выполнен ChatGPT + Vladimir in-session)

1. Бэкап: `gateway_embeddings.py.bak.stage14b`
2. Патч `gateway_embeddings.py`: загрузка первого токена из `LLM_GATEWAY_TOKENS`, отправка `Authorization: Bearer {token}` в httpx headers
3. `systemctl edit rag-mcp` → `EnvironmentFile=/home/vladimir/llm-gateway/.env`
4. `systemctl restart rag-mcp`
5. VS Code → `Developer: Reload Window` (обновление MCP streamable-http session)

### Оценка

Это **классический cascading regression**: security hardening одного компонента сломал server-to-server интеграцию с другим. Диагностирован и исправлен в рамках той же сессии. Решение прагматичное (первый токен из map) — для лаборатории достаточно. Для enterprise: dedicated service token или internal auth bypass.

## Грабли

### #48. Mandatory gateway auth ломает server-to-server RAG MCP embeddings

**Симптом:** После перехода gateway auth в mandatory, RAG MCP перестаёт работать. `search_docs` → 401 на `/v1/embeddings`.

**Причина:** `gateway_embeddings.py` (внутренний httpx client RAG MCP) не отправлял Authorization header. Дополнительно: `rag-mcp.service` не имел доступа к env с токенами.

**Классификация:** Cascading regression — security hardening одного слоя влияет на server-to-server вызовы соседнего слоя.

**Решение:** (1) Добавить Bearer auth в `gateway_embeddings.py`. (2) Добавить `EnvironmentFile=` в systemd unit для rag-mcp. (3) Reload VS Code window для обновления MCP session.

**Предотвращение:** При любом изменении auth policy — проверять ВСЕ server-to-server вызовы, не только клиентские. Чек-лист: gateway ← Continue, gateway ← orchestrator, gateway ← rag-mcp, gateway ← health-check.

## Архитектурное решение

### ADR-015: Per-user token storage — ENV-based JSON

**Контекст:** Нужно хранить маппинг token → user_id для 2–3 пользователей лаборатории.

**Варианты:** (A) ENV-based JSON в systemd EnvironmentFile; (B) Отдельный YAML config file.

**Выбор:** Вариант A — `LLM_GATEWAY_TOKENS` env.

**Обоснование:** Ноль новых файлов конфигурации, secret isolation через systemd/chmod 600, достаточно для 2–3 пользователей. При масштабировании до 5+ → миграция на config file или secrets manager.

**Последствия:** `.env` файл shared между `llm-gateway.service` и `rag-mcp.service` через `EnvironmentFile=`.

## Стратегический результат

1. **Mandatory per-user auth** — каждый запрос к GPU идентифицирован. Enterprise-паттерн: трассируемость использования ресурсов.

2. **Audit trail** — `user_id` в каждом structured log event. Парсинг одной командой jq. Готов к агрегации и алертам.

3. **Security regression обнаружена и починена** — server-to-server вызовы RAG MCP адаптированы к mandatory auth. Грабли #48 задокументированы для будущих изменений auth policy.

4. **Unified .env** — один файл с токенами, shared между gateway и rag-mcp через systemd EnvironmentFile. Простая ротация: поменять .env → restart обоих сервисов.
