## Задача

Два связанных подэтапа в одном этапе:

1. **10A-1: Модуляризация gateway.py** — разбиение монолита (1305 строк) на Python-пакет `gateway/` с ≥5 модулями. Выполнение обязательства ADR-008.

## Результат: 10A-1 (Модуляризация)

### Что было

Монолитный файл `gateway.py` v0.8.0 — 1305 строк, все 4 endpoint'а, вся бизнес-логика, Pydantic-модели, retry, error handling, auth — в одном файле.

### Что стало

Python-пакет `gateway/` с 8 модулями + entrypoint `run.py`:

| Файл | Строк | Назначение |
|------|-------|-----------|
| `run.py` | 25 | Uvicorn entrypoint: `gateway.app:app` |
| `gateway/__init__.py` | 113 | VERSION="0.9.0", все константы |
| `gateway/logging_config.py` | 106 | JSONFormatter, setup_logging() |
| `gateway/models.py` | 135 | 11 Pydantic/dataclass моделей |
| `gateway/errors.py` | 69 | GatewayError + 3 exception handlers |
| `gateway/upstream.py` | 212 | httpx client, retry (stream + non-stream), OOM classification |
| `gateway/chat.py` | 595 | /v1/chat/completions — полный цикл + structured logging |
| `gateway/embeddings.py` | 277 | /v1/embeddings — полный цикл + structured logging |
| `gateway/listing.py` | 108 | /v1/models + /health + parse_ollama_timestamp |
| `gateway/app.py` | 153 | FastAPI assembly: middleware, handlers, routers |
| **Итого** | **1793** | vs 1305 монолит (+488 на imports, docstrings, logging) |

### Архитектурные решения

- **APIRouter** в каждом модуле (chat, embeddings, listing) → `app.include_router()` в app.py. Стандартный FastAPI-паттерн.
- **Иерархия импортов** без циклических зависимостей: `__init__` → `logging_config` → `models` → `errors` → `upstream` → `chat/embeddings/listing` → `app`.
- **Один процесс** — один systemd unit, один порт. Модуляризация логическая, не сервисная.

### Обновления systemd

`llm-gateway.service` ExecStart изменён:

## Результат: 10A-2 (Structured JSON Logging)

### Архитектурные решения

| Решение | Выбор | Обоснование |
|---------|-------|-------------|
| Formatter | Custom JSONFormatter (stdlib) | Ноль внешних зависимостей, полный контроль схемы |
| Access log | Схема C — uvicorn.access отключён, свой middleware | Единая JSON-схема, LLM-доменные поля в каждом event |
| Event types | 2 типа: `llm_completion` (chat) и `llm_embedding` | Разная семантика HTTP vs LLM |
| Exception handling | traceback → строка в поле `error` | Один JSON-объект = одна строка в journal |

### Парсинг логов

```bash
# Все JSON-события:
journalctl -u llm-gateway -o cat --no-pager | jq -R 'fromjson? | select(.)'

# Только LLM chat:
journalctl -u llm-gateway -o cat --no-pager | jq -R 'fromjson? | select(.event == "llm_completion")'

# Только embeddings:
journalctl -u llm-gateway -o cat --no-pager | jq -R 'fromjson? | select(.event == "llm_embedding")'

# Медленные запросы (>5 сек):
journalctl -u llm-gateway -o cat --no-pager | jq -R 'fromjson? | select(.latency_ms? > 5000)'
```

## Грабли

### Грабля #30: systemd ExecStart placeholder

**Проблема:** В инструкции деплоя ExecStart содержал placeholder `/.../venv/bin/python` вместо реального пути. Systemd вернул status=203/EXEC.

**Решение:** Заменить на полный путь `/home/vladimir/llm-gateway/venv/bin/python run.py`.

**Урок:** Всегда давать абсолютные пути в systemd unit-файлах, не использовать placeholder'ы.

### Грабля #31: scp в несуществующую директорию

**Проблема:** `scp` файлов в `~/llm-gateway/gateway/` до создания директории `gateway/` на сервере — ошибка "No such file or directory".

**Решение:** `mkdir -p ~/llm-gateway/gateway` перед scp.

**Урок:** scp не создаёт промежуточные директории. Всегда `mkdir -p` перед массовым scp.
