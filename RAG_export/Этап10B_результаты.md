## Задача

Добавить endpoint `GET /metrics` в gateway для внешнего мониторинга — in-memory счётчики запросов, ошибок, токенов и latency по endpoint'ам и моделям. Формат: JSON (Prometheus text format отложен до этапа 15).

---

## Результат

### Новый модуль: `gateway/metrics.py`

| Компонент | Описание |
|-----------|----------|
| `MetricsCollector` | Singleton-класс с thread-safe (threading.Lock) in-memory счётчиками |
| `collector` | Глобальный экземпляр, импортируется в chat.py и embeddings.py |
| `GET /metrics` | FastAPI route, возвращает JSON-снимок всех счётчиков |

Счётчики — монотонные (monotonic counters): только растут, сбрасываются при рестарте процесса. Это by design: персистентная история — в structured logs (journalctl), метрики — для live-наблюдения.

### Метрики: три группы

**Группа 1 — Общие (totals):** total requests, total errors, uptime_seconds, started_at, gateway_version.

**Группа 2 — По endpoint'ам:** requests, errors, total_prompt_tokens, total_completion_tokens, latency_ms_sum, latency_ms_count. Средний latency вычисляется потребителем: `latency_ms_sum / latency_ms_count`.

**Группа 3 — По моделям:** requests, total_tokens, errors.

### Архитектурные решения

| Решение | Выбор | Обоснование |
|---------|-------|-------------|
| Формат ответа | JSON | Совпадает с jq-воркфлоу лаборатории. Prometheus — этап 15 |
| Интеграция | Явный вызов `collector.record(log_data)` рядом с каждым structured log | Минимальная связанность, данные гарантированно совпадают с логами |
| Thread safety | threading.Lock | Наносекунды overhead, безопасность при будущих workers > 1 |
| Auth | /metrics освобождён от Bearer (как /health) | Мониторинговый endpoint, AUTH_EXEMPT_PATHS в app.py |
| Singleton | Module-level `collector = MetricsCollector()` | Подтверждено: `chat.collector is metrics.collector == True` |

## Инцидент при деплое

### Симптом
После первоначальной заливки файлов embeddings не отражались в `/metrics`. Chat — отражался корректно.

### Root cause
На сервер был загружен старый `embeddings.py` (v0.9.0 без `from .metrics import collector`). Файл `chat.py` при этом был корректной версии 10B.

### Диагностика
```bash
python - <<'PY'
import gateway.metrics as m
import gateway.embeddings as e
print("embeddings collector is metrics collector:", e.collector is m.collector)
PY
# Результат: AttributeError: module 'gateway.embeddings' has no attribute 'collector'
```

### Причина
Deployment mismatch — при скачивании 5 файлов из Claude один файл (embeddings.py) не был обновлён или был перезаписан предыдущей версией.

### Fix
Повторная загрузка корректного `embeddings.py` → `grep -c "collector.record"` = 4 → рестарт → проблема устранена.

### Грабли #32
Класс ошибки тот же, что #30 (systemd placeholder) и #31 (scp mkdir) — deployment/file state mismatch. При замене нескольких файлов одного пакета обязательна верификация каждого файла до рестарта.

**Превентивная проверка (добавить в процедуру деплоя):**
```bash
# После scp всех файлов, до рестарта:
grep "VERSION" ~/llm-gateway/gateway/__init__.py
grep -c "collector.record" ~/llm-gateway/gateway/chat.py ~/llm-gateway/gateway/embeddings.py
```

## Observability baseline: что теперь есть

| Компонент | Этап | Формат | Назначение |
|-----------|------|--------|-----------|
| `GET /health` | 5 | JSON | Alive/dead, Ollama status |

Три компонента образуют минимальный observability baseline для backend-сервиса. Enterprise-эквивалент: health check + structured logs + metrics endpoint — стандартная тройка для любого микросервиса.
