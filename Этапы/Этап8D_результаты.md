## Что сделано

### Введение

Этап 8D формализует `orchestrator.py` как **инфраструктурный инструмент** слоя 4 (Orchestration/Automation): из интерактивного PoC он превращается в компонент, вызываемый из shell-скриптов и git-хуков без участия IDE и без человека в контуре.

Ключевой паттерн этапа: **оркестратор как backend, shell-скрипты как frontend**. Скрипты отвечают за получение контекста (git diff, содержимое файла) и форматирование задачи; оркестратор отвечает за выполнение LLM-pipeline и возврат текста.

## Архитектурные решения

### Решение 1: Контракт запуска orchestrator через venv

`orchestrator.py` использует `openai` SDK — пакет установлен только в `~/llm-gateway/venv`, не в системном Python. Все скрипты жёстко вызывают:

```bash
ORCH_PY="$ROOT_DIR/venv/bin/python"
"$ORCH_PY" "$ORCH_SCRIPT" --pipeline ... --task ... --stdout
```

Это исключает хрупкую зависимость от `PATH` и системного интерпретатора.

### Решение 2: --stdout как bridge между оркестратором и shell

Флаг `--stdout` добавлен в orchestrator.py: при его наличии текст последнего шага pipeline пишется в `sys.stdout` (чистый текст, без JSON-обёртки). Сохранение JSON в `results/` при этом продолжается. Это единственный способ передачи результата в shell-скрипты без зависимости от `jq`.

### Решение 3: commit-msg pipeline — намеренно одноступенчатый

Для генерации коммит-сообщений выбран `qwen3.5:9b` (не `qwen3-coder:30b`): задача текстовая, не code-heavy; модель быстрее загружается. Reviewer намеренно отсутствует — скорость важнее глубины проверки для commit message. Зафиксировано как осознанное архитектурное решение.

### Решение 4: pre-push хук — STRICT_REVIEW с fail-closed при non-TTY

Хук работает в двух режимах:
- **Нестрогий (default):** review выполняется, результат выводится, push продолжается.
- **STRICT_REVIEW=1 + TTY:** интерактивный вопрос; push продолжается при `y/Y`.
- **STRICT_REVIEW=1 + non-TTY (CI/CD):** `exit 1` с понятным логом. Fail-closed — безопасное поведение для неинтерактивных окружений.

Логика `-t 0` (проверка stdin на TTY) устраняет граблю первой версии, где `read` получал EOF без явного ответа.

### Решение 5: MAX_LINES=400 в скриптах review и commit-msg

Защита от превышения `num_ctx` при больших диффах. При превышении скрипт добавляет предупреждение в вывод. Жёсткий truncate по числу строк — простое и предсказуемое поведение.

## Известные ограничения (задокументированные)

### auto-docs.sh: docstring-режим возвращает обзорную документацию

Модель не всегда возвращает «код с вставленными docstrings», несмотря на инструкцию — вместо этого генерирует структурированный обзор модуля. Для строгого docstring-патча потребуется отдельный pipeline или более жёсткий prompt с примером формата. Принято как допустимое поведение для PoC.

### pre-push: STRICT_REVIEW без TTY в текущей среде

В текущей конфигурации подключения (SSH-сессия без проброса PTY к git hook) STRICT_REVIEW=1 всегда даёт fail-closed. Это корректное и безопасное поведение, принято как данность.

### auto-review --branch: использует имя ветки, не SHA

Для production-варианта более точный подход — использовать `REMOTE_SHA..LOCAL_SHA` из stdin pre-push хука. Текущая реализация через `git diff <branch>...HEAD` достаточна для PoC.

## Подтверждённые сценарии (end-to-end тест 8D-7)

| Сценарий | Команда | Результат |
|----------|---------|-----------|
| Code review (last-commit) | `./scripts/auto-review.sh --last-commit` | Review выведен в stdout, exit 0 |
| Commit message | `git add README_8D4.md && ./scripts/auto-commit-msg.sh` | `docs(README_8D4): add duplicate test 8D-4 header`, exit 0 |
| Docs generation | `./scripts/auto-docs.sh gateway.py --format markdown` | Структурированная markdown-документация gateway.py |
| pre-push non-strict | `git push origin master` | Review + `non-strict mode, push will continue`, push прошёл |
| pre-push STRICT non-TTY | `STRICT_REVIEW=1 git push origin master` | Review + `failing closed. Push aborted`, exit 1 |
| Пустой diff | `./scripts/auto-review.sh --staged` (без staged) | `No changes to review`, exit 0 |
| Без staged файлов | `./scripts/auto-commit-msg.sh` | ERROR + exit 1 |

## Грабли (зафиксированы для Obsidian → Грабли/)

1. **`python3 orchestrator.py` без venv** → `ModuleNotFoundError: openai`. Всегда вызывать через `~/llm-gateway/venv/bin/python`.
2. **`load_pipelines()` плоский словарь** → `--list` показывал `pipelines (0 steps)`. Исправлено: явное извлечение `data["pipelines"]` при наличии ключа.
3. **`read` в pre-push без проверки TTY** → EOF, неочевидное поведение. Исправлено: `-t 0` + явный fail-closed.
4. **`git diff HEAD~1 HEAD` на репозитории с одним коммитом** → ошибка git. Корректно обработано: пустой diff → `No changes to review`, exit 0.
