# ADR-xxx — commit-msg pipeline: намеренно одноступенчатый, qwen3.5:9b

## Контекст

Для генерации conventional commit message нужен pipeline в orchestrator. Вопрос: какую модель и сколько шагов использовать?

## Варианты

**A. execute-review: executor (qwen3-coder:30b) + reviewer (deepseek-r1:32b)**  
Минус: ~40–45 сек для генерации одной строки текста — несоразмерно задаче. qwen3-coder:30b оптимизирован для кода, не для форматированного текста.

**B. Один шаг, qwen3.5:9b (выбрано)**  
Плюс: ~5–10 сек, модель «повседневный chat» хорошо справляется с текстовыми задачами. Commit message — не код, а форматированный текст с жёсткими правилами (72 символа, imperative, conventional format).

**C. Один шаг, qwen3-coder:30b**  
Нейтрально: модель справится, но избыточна для задачи; занимает VRAM, которую лучше отдать под следующий шаг.

## Решение

Вариант **B**. `commit-msg` pipeline: 1 шаг, `qwen3.5:9b`, `temperature: 0.1` (детерминизм), `max_tokens: 128` (одна строка), `num_ctx: 4096`.

Reviewer намеренно отсутствует: для commit message **скорость важнее глубины проверки**. Пользователь видит результат и может скорректировать вручную.

## Последствия

- Commit-msg генерируется за ~5–10 сек vs ~40–45 сек для двухшагового pipeline.
- Подходит для `prepare-commit-msg` hook (полная автоматизация без паузы).
- При необходимости более строгой проверки — заменить на `execute-review` в конфигурации скрипта.
