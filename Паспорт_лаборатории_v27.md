## 1. Назначение лаборатории

Цель: отладка и эксплуатация локального стек-паттерна для LLM (Large Language Model — большая языковая модель) без облаков — один Ubuntu-сервер с GPU (Graphics Processing Unit — графический процессор), Ollama как менеджер моделей, OpenAI-совместимый HTTP-шлюз, подключение IDE (Integrated Development Environment — интегрированная среда разработки: VSCode + Continue.dev + Copilot BYOK) и CLI-клиентов.

Стратегия использования — **Depth over Speed**: приоритет максимальной глубины контекста и работы с тяжёлыми reasoning-моделями, используя системную ОЗУ (RAM — Random Access Memory — оперативная память) как расширение видеопамяти. Скорость генерации (tokens/sec) — вторичный приоритет.

Задачи: тестирование больших локальных моделей (Qwen, DeepSeek, GLM, Gemma и др.) под реальной инженерной нагрузкой; разработка и проверка архитектурных паттернов (шлюзы, API — Application Programming Interface, мониторинг, интеграция с Dev-инструментами); подготовка решений, переносимых на прод-серверы.

## 5. LLM-уровень: Ollama и модели

### 5.1. Ollama — конфигурация

Адрес: `http://localhost:11434` / `http://192.168.0.128:11434`. Systemd: `ollama.service`. Версия: **0.20.2**.

Override `/etc/systemd/system/ollama.service.d/override.conf`:

| Переменная | Значение | Назначение |
|---|---|---|
| `OLLAMA_HOST` | `0.0.0.0:11434` | Все интерфейсы |
| `OLLAMA_FLASH_ATTENTION` | `1` | Flash Attention (оптимизация вычисления внимания) |
| `OLLAMA_KV_CACHE_TYPE` | `q8_0` | KV-кэш (Key-Value cache) квантизация до 8 бит |
| `OLLAMA_MAX_LOADED_MODELS` | `2` | FIM + chat/agent одновременно |
| `OLLAMA_NUM_PARALLEL` | `1` | Один параллельный запрос к модели (autocomplete и chat конкурируют за GPU) |
| `CUDA_VISIBLE_DEVICES` | `0` | GPU 0 |
| `GGML_CUDA_NO_GRAPHS` | `1` | Отключение CUDA graph capture |

**Политика квантизации и KV-кэша.** Лаборатория использует только схемы квантизации, официально поддержанные в Ollama/llama.cpp. Текущая схема KV-кэша: `q8_0` (официально поддержана с Ollama 0.4+). Исследовательские схемы допускаются только как изолированный PoC (Proof of Concept — проверка концепции) без влияния на основной контур. Переход на новую схему требует: (1) официальной поддержки в используемой версии Ollama, (2) A/B-сравнения, (3) записи результата в ADR (Architecture Decision Record — документ архитектурного решения).

### 6.2. Ключевые возможности

- Настоящий стриминг (TTFT — Time To First Token = 0.12s)
- Reasoning policy (effort → think: true/false на стороне шлюза)
- Depth-over-Speed (num_ctx 8192/32768, MAX_NUM_CTX=131072 после этапа 15)
- Валидация (Pydantic), OOM (Out of Memory)-детекция (503), retry холодного старта
- OpenAI SDK совместимость (system_fingerprint, service_tier, extra="ignore", vision, tools)
- Tool_calls проброс Ollama → OpenAI формат (stream + non-stream)
- **Bearer auth (mandatory per-user, Этап 14B):** env LLM_GATEWAY_TOKENS (JSON {token: user_id}), 3 пользователя (vladimir, roadwarrior, orchestrator). /health и /metrics exempt. user_id в каждом structured log event (audit trail). gateway отказывается стартовать без токенов.
- **Embeddings endpoint** `/v1/embeddings`: OpenAI-compatible, batch support, allowlist models, staged pipeline validation, encoding_format silent coerce (9B hotfix)
- **Structured JSON logging** (Этап 10A): каждый запрос → JSON event в systemd journal. События: `llm_completion` (chat с tokens, latency_ms, ttft_ms, tool_calls_count, reasoning_effort), `llm_embedding` (embed с dim, latency_ms), `llm_orchestrate` (pipeline с steps_count, total_duration). Custom JSONFormatter (stdlib, ноль зависимостей). uvicorn.access отключён — свой middleware. Парсинг: `journalctl -o cat | jq -R 'fromjson? | select(.)'`
- **In-memory metrics** (Этап 10B): `GET /metrics` — JSON-снимок агрегированных счётчиков. Три группы: totals (requests, errors, uptime), endpoints (per-path tokens, latency_ms_sum/count), models (per-model requests/tokens/errors). Счётчики монотонные, сбрасываются при рестарте. Без auth (как /health). Thread-safe (threading.Lock)
- **Orchestration endpoint** (Этап 16): `POST /v1/orchestrate` — sequential multi-model pipeline через self-call. Каждый шаг = POST /v1/chat/completions на localhost с service token `orchestrator`. Auth, logging, metrics для каждого шага автоматически. 6 pipeline из `pipelines.yaml`. Structured log: `llm_orchestrate` event. `GET /v1/orchestrate/pipelines` — pipeline registry API.

### 7.5. Грабли Continue 1.2.22 (полный список)

**Конфигурация:**
- **timeout** — в миллисекундах (OpenAI SDK). 600000 = 10 минут для chat, 30000 = 30 сек для autocomplete
- **extraBodyProperties** — ВНУТРИ `requestOptions`, НЕ на уровне модели
- **YAML anchors** (`<<: *defaults`) — НЕ принимает
- **Имена моделей со скобками** — вызывает `Invalid input`
- **roles: agent** — невалидная роль
- **tabAutocompleteModel** (top-level) — не существует в YAML schema v1
- **debounce → debounceDelay** в YAML-схеме
- **Дефолтный таймаут autocomplete ~5 сек** — мало для холодного старта
- **Секция context:** сбрасывает дефолтные провайдеры — все перечислять явно
- **Проектные rules** — требуют `Developer: Reload Window` при первом создании
- **capabilities: [tool_use]** — НЕ требуется, Continue автодетектит
- **MCP серверы работают ТОЛЬКО в Agent mode** — не в Chat, не в Edit
- **MCP YAML format (Грабли #60):** формат docker.yaml должен совпадать с git.yaml/rag.yaml — плоская структура `type:` + `url:`, не вложенный `transport:` блок
- **MCP reconnect после SSH tunnel (Грабли #63):** Continue не reconnects автоматически при поднятии tunnel. Требуется `Developer: Reload Window`.

**Embeddings (9B):**
- **encoding_format: float обязателен** — Continue `provider: openai` отправляет несовместимый encoding_format. Fix: `requestOptions.extraBodyProperties.encoding_format: float`
- **apiKey заглушка обязательна** — Continue `provider: openai` требует непустой apiKey даже без auth. Fix: `apiKey: ollama`
- **Rebuild codebase index → EBUSY на Windows** — штатная команда не может удалить `index.sqlite`. Fix: cold restart VS Code + ручное переименование `%USERPROFILE%\.continue\index\`
- **@Codebase provider deprecated** — помечен deprecated в документации Continue; Agent mode built-in tools рекомендуется как замена. Deprecated provider даёт нестабильные результаты (synthetic code вместо grounded retrieval)

**Модели:**

### 9.1. UFW Firewall (14A)

Статус: **Active**, default deny incoming.

| # | Порт | Протокол | Доступ | Назначение |
|---|------|----------|--------|-----------|
| 1 | 22/tcp | SSH | LAN 192.168.0.0/24 + WG 10.10.10.2 | Управление |

## 13. Политика актуальности и устаревания

### 13.1. Природа документа

Паспорт фиксирует **снимок состояния** лаборатории на дату конкретной версии. Любое расхождение между фактическим стеком и содержимым паспорта считается **техническим долгом**.

### 13.2. Правило ревизии версий

Обязательная ревизия при: смене версии ключевого компонента на 2+ minor, компоненте старше 6 месяцев, завершении этапа, новой версии архитектуры.

Ключевые компоненты: Ubuntu, NVIDIA Driver, CUDA, Docker Engine, Ollama, gateway.py, Continue.dev, VS Code, uv/uvx.

### 13.3. Компоненты со статусами PoC, Legacy, Deprecated

- **PoC** пересматриваются раз в квартал.
- **Legacy** требуют плана замены (2 квартала → Deprecated).
- **Deprecated** выводятся до следующей плановой ревизии.
