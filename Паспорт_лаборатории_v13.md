Является снимком реализации целевой архитектуры (source of truth — единого источника правды). Любое расхождение между паспортом и фактическим стеком считается техническим долгом. Любое расхождение между паспортом и целевой архитектурой — либо устраняется, либо оформляется явно как архитектурное исключение.

## 1. Назначение лаборатории

Цель: отладка и эксплуатация локального стек-паттерна для LLM (Large Language Model — большая языковая модель) без облаков — один Ubuntu-сервер с GPU (Graphics Processing Unit — графический процессор), Ollama как менеджер моделей, OpenAI-совместимый HTTP-шлюз, подключение IDE (Integrated Development Environment — интегрированная среда разработки: VSCode + Continue.dev + Copilot BYOK) и CLI-клиентов.

Стратегия использования — **Depth over Speed**: приоритет максимальной глубины контекста и работы с тяжёлыми reasoning-моделями, используя системную ОЗУ (RAM — Random Access Memory — оперативная память) как расширение видеопамяти. Скорость генерации (tokens/sec) — вторичный приоритет.

Задачи: тестирование больших локальных моделей (Qwen, DeepSeek, GLM и др.) под реальной инженерной нагрузкой; разработка и проверка архитектурных паттернов (шлюзы, API — Application Programming Interface, мониторинг, интеграция с Dev-инструментами); подготовка решений, переносимых на прод-серверы.

## 5. LLM-уровень: Ollama и модели

### 5.1. Ollama — конфигурация

Адрес: `http://localhost:11434` / `http://192.168.0.128:11434`. Systemd: `ollama.service`. Версия: **0.18.0**.

Override `/etc/systemd/system/ollama.service.d/override.conf`:

| Переменная | Значение | Назначение |
|---|---|---|
| `OLLAMA_HOST` | `0.0.0.0:11434` | Все интерфейсы |
| `OLLAMA_FLASH_ATTENTION` | `1` | Flash Attention (оптимизация вычисления внимания) |
| `OLLAMA_KV_CACHE_TYPE` | `q8_0` | KV-кэш (Key-Value cache) квантизация до 8 бит |
| `OLLAMA_MAX_LOADED_MODELS` | `2` | FIM + chat/agent одновременно |
| `CUDA_VISIBLE_DEVICES` | `0` | GPU 0 |
| `GGML_CUDA_NO_GRAPHS` | `1` | Отключение CUDA graph capture |

**Политика квантизации и KV-кэша.** Лаборатория использует только схемы квантизации, официально поддержанные в Ollama/llama.cpp. Текущая схема KV-кэша: `q8_0` (официально поддержана с Ollama 0.4+). Исследовательские схемы (TurboQuant и аналоги) допускаются только как изолированный PoC (Proof of Concept — проверка концепции) без влияния на основной контур. Переход на новую схему требует: (1) официальной поддержки в используемой версии Ollama, (2) A/B-сравнения, (3) записи результата в ADR (Architecture Decision Record — документ архитектурного решения).

### 7.5. Грабли Continue 1.2.17 (полный список)

**Конфигурация:**
- **timeout** — в миллисекундах (OpenAI SDK). 600000 = 10 минут для chat, 30000 = 30 сек для autocomplete
- **extraBodyProperties** — ВНУТРИ `requestOptions`, НЕ на уровне модели
- **YAML anchors** (`<<: *defaults`) — НЕ принимает
- **Имена моделей со скобками** — вызывает `Invalid input`
- **roles: agent** — невалидная роль
- **tabAutocompleteModel** (top-level) — не существует в YAML schema v1
- **debounce → debounceDelay** в YAML-схеме
- **Дефолтный таймаут autocomplete ~5 сек** — мало для холодного старта
- **Секция context:** сбрасывает дефолтные провайдеры — все перечислять явно
- **Проектные rules** — требуют `Developer: Reload Window` при первом создании
- **capabilities: [tool_use]** — НЕ требуется, Continue автодетектит
- **MCP серверы работают ТОЛЬКО в Agent mode** — не в Chat, не в Edit

**Модели:**
- **qwen3.5:35b** — НЕ поддерживает function calling в Ollama (НИКОГДА для Agent), НЕ использовать как Planner в orchestrator
- **qwen3-coder:30b** — надёжный Agent, работает с 26+ tools (12 MCP + 14 built-in)

**Windows / Terminal (Continue Agent):**
- **PowerShell-native обязателен** — Agent по умолчанию может генерировать bash-style (`ls -la`); фиксируется через 01-general.md (alwaysApply)
- **UTF-8 preamble** — для корректного вывода кириллических имён файлов необходимо выставлять кодировки в начале команды: `$OutputEncoding = [System.Text.UTF8Encoding]::new(); [Console]::InputEncoding = [System.Text.UTF8Encoding]::new(); [Console]::OutputEncoding = [System.Text.UTF8Encoding]::new();`
- **Occasional quoted-command** — редкий дефект: модель иногда оборачивает команду в строку и печатает вместо выполнения; не блокирующий, known issue качества tool-call generation
- **Reload Window + новый чат** — обязательный цикл проверки при изменении rules

## 8. Слой 4: Orchestration / Automation

### 8.1. orchestrator.py (Этап 8C) — Active

**Расположение:** `~/llm-gateway/orchestrator.py`  
**Среда выполнения:** Python venv на сервере Ubuntu, `~/llm-gateway/`  
**Вызывает:** gateway.py `/v1/chat/completions` через OpenAI SDK (`extra_body` для `num_ctx`, `temperature`)

**Назначение:** Sequential multi-model pipeline. Каждый шаг = отдельный запрос к gateway.py с системным промптом своей роли и контекстом (результатом) предыдущего шага.

**Реализованные pipeline (`~/llm-gateway/pipelines.yaml`):**

| Pipeline | Шаги | Модели | Типичное время |
|----------|------|--------|---------------|
| `smoke` | smoke | qwen3.5:9b | ~5 сек |
| `execute-review` | executor → reviewer | qwen3-coder:30b → deepseek-r1:32b | ~40–45 сек |
| `plan-execute-review` | planner → executor → reviewer | qwen3:30b → qwen3-coder:30b → deepseek-r1:32b | 53–88 сек |
| `review-only` | reviewer | deepseek-r1:32b | ~35–37 сек |
| `docs-generate` | executor | qwen3-coder:30b | ~5–10 сек |

**Реальные замеры времени на задачах (pipeline `plan-execute-review`):**

| Task level | Planner, сек | Executor, сек | Reviewer, сек | Total, сек |
|------------|-------------|--------------|--------------|-----------|
| Simple (email validation) | 45.25 | ~6 | ~37 | **88.49** |
| Medium (async/await + aiohttp) | 11.80 | 4.08 | 37.46 | **53.34** |
| Complex (log errors by type) | 26.63 | 5.01 | 35.02 | **66.66** |

**Ключевые технические особенности:**
- Фильтрация `<think>...</think>` блоков по имени шага `planner` (qwen3:30b генерирует внутренний монолог)
- Результаты сохраняются в `~/llm-gateway/results/YYYYMMDD_HHMMSS.json`
- `temperature: 0.2` для Executor и Reviewer; `temperature: 0.4` для Planner; `num_ctx: 8192`

**Известное ограничение:** qwen3.5:35b нестабилен как Planner (500/502 при загрузке в Ollama 0.18.0 на RTX 3090). Planner = `qwen3:30b` — надёжная рабочая замена.

## 12. Политика актуальности и устаревания

### 12.1. Природа документа

Паспорт фиксирует **снимок состояния** лаборатории на дату конкретной версии. Он не является постоянной истиной — стек меняется, и паспорт должен следовать за ним.

Любое расхождение между фактическим стеком и содержимым паспорта считается **техническим долгом**. Его необходимо либо устранить (обновить паспорт), либо зафиксировать явно с указанием причины и горизонта исправления.

### 12.2. Правило ревизии версий

Обязательная ревизия и обновление паспорта инициируется при любом из следующих условий:

- Установленная версия ключевого компонента отличается от указанной в паспорте **более чем на 2 минорных релиза**.
- Компонент используется в версии **старше 6 месяцев** от даты выхода более новой стабильной версии.
- Завершён новый этап разработки, изменивший состав компонентов или их конфигурацию.
- Вышла новая версия целевой архитектуры, затрагивающая данный стенд.

Ключевые компоненты для отслеживания версий: Ubuntu, NVIDIA Driver, CUDA, Docker Engine, Ollama, gateway.py, Continue.dev, VS Code, uv/uvx.

### 12.3. Компоненты со статусами PoC, Legacy, Deprecated

- **PoC-компоненты** пересматриваются **раз в квартал**: либо переводятся в Active, либо удаляются.
- **Legacy-компоненты** требуют плана замены. При отсутствии плана в течение 2 кварталов переводятся в Deprecated.
- **Deprecated-компоненты** должны быть выведены из эксплуатации до следующей плановой ревизии паспорта.

### 12.4. Политика квантизации и KV-кэша в контексте паспорта

При изменении схемы квантизации (KV-кэш или веса модели) в основном контуре:
1. Обновить соответствующую строку в таблице 5.1 (Ollama — конфигурация).
2. Зафиксировать обоснование в ADR или комментарии к изменению.
3. Актуализировать метрики производительности в секции 4.1 целевой архитектуры.

## 13. Регулярная ревизия паспорта

### 13.1. Периодичность

Плановая ревизия — **раз в 3 месяца**. Внеплановая — по триггерам из раздела 12.2.

### 13.2. Связка с целевой архитектурой

- При изменении **целевой архитектуры** → выпускается новая минорная версия паспорта (не позднее 2 недель после публикации архитектуры).
- При изменениях **в паспорте**, не согласованных с архитектурой → требуется ADR или явное архитектурное решение.

### 13.3. Ответственность

Владелец лаборатории несёт ответственность за актуальность паспорта. При передаче лаборатории новому владельцу — первое действие нового владельца: пройти чек-лист актуальности (раздел 14).
