## Что сделано

### Введение

Этап 8C реализует первый компонент слоя 4 (Orchestration/Automation) целевой архитектуры: `orchestrator.py` — Python-скрипт, выполняющий последовательный multi-model pipeline через gateway.py по OpenAI-совместимому API. Скрипт работает автономно на сервере Ubuntu, без участия IDE и Continue.dev — это самостоятельный headless-инструмент.

Ключевой паттерн этапа: **Planner → Executor → Reviewer**. Каждая роль в pipeline отдана специализированной модели; результат каждого шага передаётся следующей модели как контекст. Это позволяет получить качество, недостижимое при использовании одной модели: Planner думает о структуре, Executor реализует, Reviewer критически анализирует и улучшает.

### Реализованные pipeline

| Pipeline | Шаги | Модели |
|----------|------|--------|
| `smoke` | smoke | qwen3.5:9b |
| `execute-review` | executor → reviewer | qwen3-coder:30b → deepseek-r1:32b |
| `plan-execute-review` | planner → executor → reviewer | qwen3:30b → qwen3-coder:30b → deepseek-r1:32b |
| `review-only` | reviewer | deepseek-r1:32b |
| `docs-generate` | executor | qwen3-coder:30b |

Все pipeline полностью отработали на реальном железе. `smoke` — базовый дымовой тест (Hello World). `execute-review` — стандартный рабочий контур для задач с готовой спецификацией. `plan-execute-review` — полный трёхшаговый pipeline для сложных задач.

### Ключевые технические решения

**Передача контекста между шагами:** каждый следующий шаг получает `messages` с системным промптом своей роли + результат предыдущего шага как `user`-сообщение. Паттерн универсален и не зависит от формата модели.

**Фильтрация `<think>...</think>`:** qwen3:30b в режиме Planner генерирует внутренний монолог в `<think>`-блоках перед ответом. Orchestrator фильтрует эти блоки по имени шага `planner` перед передачей в Executor — иначе Executor получил бы «план + внутренний монолог» в контексте.

**Параметры запроса:** `num_ctx: 8192`, `temperature: 0.2` для Executor и Reviewer (детерминированность), `temperature: 0.4` для Planner (небольшая вариативность в планировании). Параметры передаются через `extra_body` OpenAI SDK.

**Сохранение результатов:** каждый pipeline-запуск сохраняет JSON-артефакт в `~/llm-gateway/results/YYYYMMDD_HHMMSS.json` с полями `task`, `pipeline`, шагами (`model`, `duration_sec`, `output`) и `total_duration_sec`.

**Наблюдение по замерам.** Planner (qwen3:30b, 18 ГБ) вариативен по времени: Simple занял 45 сек (вероятно, cold start + развёрнутый план), Medium — 12 сек (модель уже в памяти, краткий план). Executor (qwen3-coder:30b) стабильно быстр: 4–6 сек на генерацию кода, что говорит о компактных, конкретных выводах. Reviewer (deepseek-r1:32b) стабильно занимает 35–37 сек — модель последовательно применяет reasoning перед ответом.

Суммарное время pipeline: 53–88 сек — существенно меньше изначальной оценки 3–8 минут. Холодный старт при смене модели укладывается в ~10–15 сек и уже включён в `duration_sec` каждого шага.

## Риски и ограничения

### Qwen3.5:35b нестабилен как Planner (критический)

Попытка использовать `qwen3.5:35b` в роли Planner в pipeline `plan-execute-review` завершилась ошибкой: gateway возвращал 500/502, логи Ollama: `model failed to load` / `llama runner terminated`. Причина — нестабильность модели при загрузке под Ollama 0.18.0 на RTX 3090 при текущих параметрах (`OLLAMA_NUM_PARALLEL=1`, KV cache `q8_0`).

**Решение:** Planner переключён на `qwen3:30b`. Модель стабильна, вписывается в VRAM при текущей квантизации и демонстрирует качественное планирование. `qwen3.5:35b` остаётся в статусе **Active** для экспериментального chat, но **не используется в orchestrator** до отдельного PoC и, возможно, обновления Ollama.

### OLLAMA_NUM_PARALLEL=1 — только последовательное выполнение

Ограничение инфраструктуры: pipeline выполняется строго последовательно, без параллелизма между шагами. Это соответствует стратегии Depth over Speed — принимается как данность на текущем стенде.

### Валидация и error handling в orchestrator (low priority)

Глубокая валидация входных данных (форматы email, лог-форматы, HTTP-статусы) реализована в коде Executor и Reviewer на уровне их рекомендаций — не в самом скрипте orchestrator.py. Orchestrator фокусируется на pipeline coordination, а не на бизнес-логике задач.

## Выводы

Этап 8C **успешно завершён**. Паттерн Planner → Executor → Reviewer стабильно работает на реальном железе (RTX 3090) с реальными задачами. Три pipeline (`smoke`, `execute-review`, `plan-execute-review`) полностью отработали, результаты сохранены в JSON.
