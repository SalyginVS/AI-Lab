## Задача

Подключить Docker Engine на сервере как MCP tool для Continue Agent mode. Третий MCP-сервер в стеке (после git и RAG), второй custom, второй серверный (Ubuntu-side). Реализовать server-side policy enforcement (ADR-017) — MCP-сервер проверяет каждую операцию по YAML-policy до вызова Docker API.

---

## Стек

- **MCP SDK:** mcp 1.27.0 (FastMCP, `mcp.server.fastmcp`)
- **Docker SDK:** docker 7.1.0 (Python, sync API)
- **Docker Engine:** 28.2.2 (Ubuntu 24.04 apt package)
- **NVIDIA Container Toolkit:** 1.19.0 (GPU passthrough подтверждён)
- **Транспорт:** streamable-http (порт 8200, bind 127.0.0.1)
- **Access model:** SSH tunnel only (не LAN-visible)
- **Deployment:** systemd (`docker-mcp.service`)
- **Policy:** docker-policy.yaml (YAML-based, загрузка при старте)

---

## Архитектурные решения

### ADR-017: Docker MCP Policy Model (Canonical)

**Контекст:** Docker — инструмент с высоким blast radius. Неограниченный доступ к Docker API позволяет удалить контейнеры, образы, выполнять произвольные команды. Существующие MCP-серверы (git, RAG) не требовали server-side policy enforcement.

**Решение:** Defense in Depth — три слоя защиты:

| Слой | Где | Что контролирует |
|------|-----|-----------------|
| 1. Continue Agent approval | Клиент (Windows) | Показ tool call, ожидание подтверждения |
| 2. MCP Server policy engine | Сервер (Ubuntu) | Проверка по docker-policy.yaml |
| 3. 03-security.md rule | Клиент (Continue) | Глобальные ограничения Agent mode |

**Категории tools:** READ (всегда разрешены), LIFECYCLE (разрешены + audit), EXEC (allowlist/denylist + audit).

**Явно исключены:** docker run, build, rm, rmi, pull, push, volume rm, network rm, prune.

### Operational decision: local-only bind + SSH tunnel

**Решение:** Docker MCP server биндится **только** на 127.0.0.1:8200. Доступ с клиента — через SSH tunnel. Порт 8200 **не** открывается в LAN, **не** добавляется в UFW.

**Обоснование:** Минимальный blast radius. Docker MCP — единственный MCP-сервер с destructive capabilities (exec, lifecycle). В отличие от RAG MCP (read-only поиск) и gateway (auth protected API), Docker MCP напрямую управляет инфраструктурой. SSH tunnel обеспечивает encrypted channel + authentication без дополнительной конфигурации.

**Следствие:** RAG MCP (8100) и gateway (8000) остаются LAN-visible с UFW rules. Docker MCP (8200) — отдельная security zone: local-only + SSH.

### 12-2. MCP Server — код ✅

Реализованы ChatGPT + Vladimir:

**server.py** — FastMCP server, 10 tools + policy engine. Async tool handlers с sync Docker SDK через `asyncio.to_thread()`. Runtime: `uvicorn.run(mcp.streamable_http_app(), host="127.0.0.1", port=8200)`.

**policy.py** — загрузка docker-policy.yaml, enforcement: `check_category()`, `check_blocked_operation()`, `check_exec_command()` с raise-методами.

**Реализованные tools (10):**

| # | Tool | Категория | Docker API |
|---|------|-----------|------------|
| 1 | `system_info` | READ | `client.df()` → containers/images/volumes counts |
| 2 | `list_containers` | READ | `client.containers.list(all=True)` |
| 3 | `list_images` | READ | `client.images.list()` |
| 4 | `container_inspect` | READ | `container.attrs` |
| 5 | `container_logs` | READ | `container.logs(tail=N)` |
| 6 | `container_stats` | READ | `container.stats(stream=False)` |
| 7 | `start_container` | LIFECYCLE | `container.start()` |
| 8 | `stop_container` | LIFECYCLE | `container.stop()` |
| 9 | `restart_container` | LIFECYCLE | `container.restart()` |
| 10 | `exec_command` | EXEC | `container.exec_run(cmd)` |

**Контрактная правка system_info (Грабли #61):** Docker SDK `client.df()` возвращает списки (Containers, Images, Volumes), а не числовые counters. Исправлено: `len()` по каждому списку → стабильный контракт для MCP-клиента.

## Грабли

### #59. FastMCP API drift: host/port not in run()

**Симптом:** `mcp.run(transport="streamable-http", host="127.0.0.1", port=8200)` → `TypeError: unexpected keyword argument 'host'`.

**Факт:** mcp 1.27.0 `FastMCP.run()` не принимает host/port. `streamable_http_app()` тоже без параметров (mount_path не поддерживается).

**Решение:** Явный запуск через uvicorn: `uvicorn.run(mcp.streamable_http_app(), host="127.0.0.1", port=8200)`.

**Урок:** При фиксированной версии MCP SDK (`mcp==1.27.0`) опираться только на фактическую сигнатуру, не на примеры из документации более новых версий. API FastMCP меняется между minor releases.

### #60. docker.yaml YAML format drift

**Симптом:** Continue не обнаруживал Docker MCP tools. UI показывал `Tools 0 / fetch failed`.

**Причина:** Первоначальный docker.yaml был создан с вложенным `transport:` блоком. Рабочие конфиги (git.yaml, rag.yaml) используют плоскую структуру: `type:` и `url:` на уровне элемента списка.

**Решение:** Привести docker.yaml к формату git.yaml/rag.yaml:
```yaml
mcpServers:
  - name: docker
    type: streamable-http
    url: http://localhost:8200/mcp
```

**Предотвращение:** Перед созданием нового MCP config всегда сверять формат с рабочими файлами текущей версии Continue.

### #61. system_info Docker SDK contract mismatch

**Симптом:** system_info возвращал некорректные данные — Docker SDK `client.df()` содержит списки (Containers, Images, Volumes), а не числовые счётчики.

**Решение:** Ввести `len()` для подсчёта: `containers_count`, `images_count`, `volumes_count`.

### #62. 421 Invalid Host header при обращении через LAN IP

**Симптом:** `curl http://192.168.0.164:8200/mcp` → `421 Misdirected Request, Invalid Host header`.

**Причина:** uvicorn/Starlette при bind на `127.0.0.1` отвергает запросы с Host header, не совпадающим с loopback.

**Следствие:** Подтверждает правильность operational decision: local-only + SSH tunnel. LAN-visible path через SSH tunnel на IP клиента — нерабочий без server-side доработки.

### #63. SSH tunnel runtime inconsistency

**Симптом:** `curl http://localhost:8200/mcp` → `Failed to connect` при наличии docker.yaml с `url: http://localhost:8200/mcp`.

**Причина:** В момент тестирования SSH tunnel не был реально поднят (предыдущий tunnel был на другом IP-адресе).

**Решение:** Перед проверкой Continue всегда подтверждать tunnel: `netstat -ano | findstr :8200` → `LISTENING`.

**Урок:** Continue MCP reconnect не происходит автоматически при поднятии tunnel. Требуется `Developer: Reload Window` после того, как endpoint стал реально доступен.

### #64. Port 8200 address already in use при systemd start

**Симптом:** `docker-mcp.service` в crash loop. Journal: `Errno 98 address already in use`.

**Причина:** Ручной foreground-процесс `python server.py` не был остановлен перед `systemctl start`.

**Решение:** Остановить foreground-процесс, restart service.

**Предотвращение:** Стандартная процедура: при переходе от foreground к systemd — `Ctrl+C` foreground → `systemctl start`.

## Стратегический результат

1. **Server-side policy enforcement доказан.** Паттерн: FastMCP tool handler проверяет YAML policy до вызова backend API. Опасные команды блокируются на сервере, не полагаясь на клиент. Enterprise takeaway: тиражируется на Kubernetes API, cloud CLI, database admin.

2. **Defense in Depth реализован.** Три слоя: Continue Agent approval → server policy → global security rule. Любой слой может отказать независимо.

3. **local-only + SSH tunnel как security pattern.** Для инструментов с высоким blast radius — bind на loopback, доступ через SSH. Не всё должно быть LAN-visible. Отличается от RAG MCP (read-only → LAN OK) и gateway (auth protected → LAN OK).

4. **Паттерн серверного MCP полностью тиражируем.** Третий MCP-сервер подтверждает: FastMCP + streamable-http + systemd + uvicorn — стабильная формула. Docker MCP добавляет policy layer как расширение паттерна.

5. **Tool count 27+ (12 git + 5 rag + 10 docker) — модели справляются.** gemma4:31b и qwen3-coder:30b корректно выбирают tools из расширенного набора.

6. **Грабли #59–64 задокументированы.** Ключевой урок: client-side integration (YAML format, tunnel state, VS Code reload) создаёт больше проблем, чем server-side code.
