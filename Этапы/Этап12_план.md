## Задача

Подключить Docker Engine на сервере (Ubuntu 24.04) как MCP tool для Continue Agent mode. Третий MCP-сервер в стеке (после git и RAG), второй custom, второй серверный (Ubuntu-side).

**Ключевое отличие от предыдущих MCP:** Docker — инструмент с высоким blast radius. Реализуем **server-side policy enforcement** (ADR-017): MCP-сервер проверяет каждую операцию по YAML-policy до вызова Docker API. Клиент не является единственным барьером безопасности.

## Архитектура

### Defense in Depth — 3 слоя защиты

| Слой | Где | Что контролирует |
|------|-----|-----------------|
| 1. Continue Agent approval | Клиент (Windows) | Показ tool call, ожидание подтверждения |
| 2. MCP Server policy engine | Сервер (Ubuntu) | Проверка по docker-policy.yaml |
| 3. 03-security.md rule | Клиент (Continue) | Глобальные ограничения Agent mode |

### Набор tools (10 штук)

| # | Tool | Категория | Docker-эквивалент | Policy |
|---|------|-----------|-------------------|--------|
| 1 | `list_containers` | READ | `docker ps -a` | Всегда разрешён |
| 2 | `list_images` | READ | `docker images` | Всегда разрешён |
| 3 | `container_logs` | READ | `docker logs --tail N` | tail ≤ 500 (policy limit) |
| 4 | `container_inspect` | READ | `docker inspect` | Всегда разрешён |
| 5 | `container_stats` | READ | `docker stats --no-stream` | timeout 10s (policy limit) |
| 6 | `system_info` | READ | `docker system df` | Всегда разрешён |
| 7 | `start_container` | LIFECYCLE | `docker start` | Разрешён, audit log |
| 8 | `stop_container` | LIFECYCLE | `docker stop` | Разрешён, audit log |
| 9 | `restart_container` | LIFECYCLE | `docker restart` | Разрешён, audit log |
| 10 | `exec_command` | EXEC | `docker exec` | allowlist/denylist в policy, audit log |

### Явно НЕ включены в v1

`docker run`, `docker build`, `docker rm`, `docker rmi`, `docker pull`, `docker push`, `docker volume rm`, `docker network rm`, `docker system prune`. Добавление — через дополнение к ADR-017.

### Технический стек

| Аспект | Решение |
|--------|---------|
| MCP SDK | FastMCP (`mcp` package, текущая версия — уточнить через Perplexity) |
| Docker API | Python `docker` SDK (не subprocess) |
| Транспорт | streamable-http, порт 8200 (ADR-014) |
| Auth | Bearer token через .env (ADR-015, паттерн RAG MCP) |
| Policy | docker-policy.yaml, загрузка при старте |
| Logging | structured JSON, `logging.getLogger("uvicorn.error")` |
| systemd | docker-mcp.service, After=docker.service |
| UFW | 8200/tcp — LAN (192.168.0.0/24) + WireGuard (10.10.10.0/24) |

## Известные риски

| Риск | Вероятность | Mitigation |
|------|-------------|------------|
| sync Docker SDK в async FastMCP → event loop block | Средняя | Perplexity briefing, asyncio.to_thread() если нужен |
| Docker socket permission в systemd user context | Низкая | vladimir в docker group, проверить на 12-3 |
| Модель вызывает tools не по назначению (hallucinated params) | Средняя | Policy engine отклонит невалидные операции |
| Tool count >30 (12 git + 5 rag + 10 docker + ~14 built-in) | Низкая | gemma4:31b справлялся с 26+ tools на Этапе 8A |
| docker compose отсутствует | Минимальная | Не требуется для scope v1 |
