# ADR-xxx — Запуск orchestrator.py только через venv/bin/python

## Контекст

`orchestrator.py` использует `openai` SDK, который установлен только в `~/llm-gateway/venv` — том же окружении, что использует `gateway.py` через systemd. Системный Python3 пакета не имеет.

Для shell-скриптов (auto-review.sh и др.) нужен стабильный контракт вызова.

## Варианты

**A. Установить `openai` глобально (apt/pip3 --system)**  
Минус: загрязняет системный Python, конфликты версий при обновлении, нарушает изоляцию.

**B. Использовать venv/bin/python (выбрано)**  
Плюс: явная изоляция, нет конфликтов, соответствует тому, как работает gateway.py. Минус: скрипты привязаны к конкретному пути.

**C. Установить `uv` и использовать `uv run`**  
Плюс: более современный подход. Минус: добавляет зависимость, uv уже используется только на клиенте для MCP; усложняет без выгоды на текущем PoC.

## Решение

Вариант **B**. Все скрипты используют:

```bash
ROOT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
ORCH_PY="$ROOT_DIR/venv/bin/python"
```

Путь вычисляется относительно расположения скрипта — переносимо при переносе `~/llm-gateway/` в другое место.

## Последствия

- При пересоздании venv (обновление зависимостей) скрипты продолжают работать без изменений.
- Enterprise-перенос: в production-варианте оркестратор заменяется HTTP-вызовом к gateway API — зависимость от venv исчезает.
- Документировать в Runbook: перед деплоем скриптов проверить наличие venv.
