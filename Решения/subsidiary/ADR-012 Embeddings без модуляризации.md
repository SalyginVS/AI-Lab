# ADR-008 — Embeddings: additive change без модуляризации gateway.py

## Контекст

gateway.py v0.7.0 содержит 994 строки. Целевая архитектура (v1.4) предполагала модуляризацию «при первом расширении (этап 9A или 10A)». Этап 9A добавляет `/v1/embeddings` — первое расширение после v0.7.0.

## Решение

**Реализовать 9A как additive change в текущем монолите.** Новые сущности добавляются в конец файла, chat-path не затрагивается, модуляризация переносится на [[Этапы/Этап010A_logging|10A]].

## Аргументы за

- Scope 9A — один endpoint. Модуляризация требует перестройки всего файла: создание пакета, разбивка на модули, обновление systemd, тесты импортов. Это отдельный этап, не боковой эффект.
- 10A (structured logging) затрагивает **весь** gateway.py — каждый handler, каждый response path. Это естественная точка для одновременной модуляризации: одно большое изменение вместо двух средних.
- Аддитивные изменения не нарушают обратную совместимость и не создают риска для chat-path.

## Аргументы против

- gateway.py вырастет до ~1100–1200 строк. Читаемость снижается.
- Технический долг откладывается.

## Последствия

- gateway.py v0.8.0 = монолит с embeddings-path в конце файла.
- **Обязательно в 10A:** при добавлении structured logging провести модуляризацию одновременно. Откладывать дальше 10A нельзя.
- Docstring файла частично устарел (упоминает `MAX_LOADED_MODELS=1`) — исправить при модуляризации.
