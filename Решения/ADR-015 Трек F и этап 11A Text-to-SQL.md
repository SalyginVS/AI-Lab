# ADR-011 Трек F (AI-as-Interface) и этап 11A (Text-to-SQL PoC)

## Статус

Принято. Зафиксировано в целевой архитектуре v1.6.

## Контекст

Стратегический анализ «ИИ поглощает интерфейс» ([[AI_as_Interface_Lab_Playbook]]) выявил, что лаборатория покрывает 4 из 5 enterprise-паттернов через существующий roadmap (MCP-коннекторы, LLM-оркестратор, гибридный RAG, детерминированная оркестрация + HITL). Единственный пробел — паттерн Text-to-SQL с семантическим слоем.

Вопрос: добавлять ли новый этап и трек в roadmap, или решать ad-hoc при необходимости?

## Решение

**Добавить этап 11A (Text-to-SQL PoC) и формальный Трек F (AI-as-Interface) в целевую архитектуру.**

### Этап 11A

- **Место:** между 9B✅ и 11 (RAG MCP). Зависимость: 9B✅.
- **Стек:** qwen3-coder:30b + SQLite + gateway /v1/chat/completions. Без нового железа.
- **Scope:** 20 тестовых вопросов, execution accuracy ≥ 70%, read-only, показ SQL перед выполнением.
- **Трудоёмкость:** 1 спринт.

### Трек F

```
Трек F (AI-as-Interface):  9B✅ → [11A Text-to-SQL] → [11 RAG MCP] → [AI Report]
```

Трек F — не отдельная дорожка разработки, а надстройка над треками B (Backend) и A (MCP Tools). Этапы используют существующие модели, gateway и инфраструктуру — добавляется только прикладной слой.

## Обоснование

1. **Закрытие пробела.** 5/5 enterprise-паттернов покрыты. Text-to-SQL — наиболее востребованный паттерн для ONDO enterprise (энергетика, гостиницы, производство — всё data-heavy).

2. **Benchmark для вендоров.** Лабораторный результат Text-to-SQL на реальных схемах — конкретный ответ на вендорское «ИИ заменит BI-аналитика». Без benchmark оценка вендорских заявлений остаётся субъективной.

3. **Минимальные зависимости.** 11A не требует: нового железа, новых моделей, изменений в gateway, завершения 10A/10B. Только qwen3-coder:30b (уже Active) + SQLite (stdlib Python).

4. **Логичное место в roadmap.** 11A стоит перед 11 (RAG MCP) — и потенциально даёт данные о quality qwen3-embedding для семантического слоя Vanna, что полезно для проектирования RAG pipeline.

5. **Enterprise-переносимость.** Паттерн NL→SQL + semantic layer переносится на любые СУБД и модели. Лабораторный код → enterprise без перепроектирования архитектуры.

## Альтернативы, которые были отвергнуты

- **Не добавлять 11A, решать Text-to-SQL ad-hoc.** Отвергнуто: ad-hoc эксперимент не даёт формального benchmark, не фиксирует критерии, не создаёт переносимого артефакта.
- **Начинать сразу с Vanna + ChromaDB.** Отвергнуто: premature complexity. Сначала baseline без семантического слоя (schema в system prompt), затем измерить прирост от Vanna.
- **Встроить Text-to-SQL в этап 11 (RAG MCP).** Отвергнуто: разные scope. 11 — RAG по документам (MCP server, ChromaDB, chunking). 11A — NL→SQL (direct LLM, read-only DB). Смешивание размоет оба.

- Вопрос 16: Vanna vs dbt как семантический слой → закрыть до enterprise-деплоя
