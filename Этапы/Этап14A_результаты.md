**Выполненные изменения:**

2. Добавлено узкое правило `192.168.0.164 → 11434/tcp` (только Windows client)

### 14A-2. Nightly Health-Check Cron ✅

- `/etc/sudoers.d/health-check`: `vladimir ALL=(ALL) NOPASSWD: /usr/sbin/ufw status`
- `visudo -c`: parsed OK
- Cron: `0 3 * * *` (06:00 Kyiv) → `~/llm-gateway/logs/nightly-health.log`
- Тестовый прогон: 11/11 записан в лог с timestamp и разделителем

### 14A-3. Benchmark History Wrapper ✅

**Файл:** `~/llm-gateway/scripts/benchmark-with-history.sh`

Две инженерные находки при создании:

1. **`benchmark.py --output` = directory, не file** — wrapper адаптирован
2. **Дневные baseline перезаписываются** — один файл `baseline_YYYY-MM-DD.json` на день. Pre-upgrade и post-upgrade в один день → перезапись (Грабли #46)

**Решение:** Wrapper создаёт immutable snapshot-копии: `benchmark_<label>_<timestamp>.json` и `.md`. History записывается в `~/llm-gateway/benchmarks/history.jsonl`.

**Transitional state:** Первые 2 записи в history.jsonl указывают на baseline, новые — на snapshot. Snapshot-схема считается canonical.

### 14A-4. Ollama Canary Upgrade SOP ✅

**Файл:** `~/llm-gateway/docs/SOP_Ollama_Upgrade.md`

6 шагов: pre-upgrade baseline → upgrade → post-upgrade smoke → post-upgrade benchmark → rollback → финализация. Критерии rollback: tok/s деградация > 20%, TTFT > 30%, любой pass→fail.

### 14A-5. Фактический Ollama Upgrade ✅

В ходе проверки SOP выполнен реальный upgrade:
- **Ollama 0.20.0 → 0.20.2**
- health-check 11/11 после upgrade
- benchmark snapshot создан (`post-upgrade-0.20.2`)
- gateway, embeddings, RAG MCP — без деградации

## Грабли

### #46. benchmark.py baseline перезаписывается при повторном запуске

**Симптом:** `benchmark.py --output benchmarks/` создаёт файл `baseline_YYYY-MM-DD.json`. Два запуска в один день → второй перезаписывает первый.

**Влияние:** Canary upgrade SOP (pre-upgrade → post-upgrade) в рамках одного дня теряет pre-upgrade данные.

**Решение:** `benchmark-with-history.sh` создаёт immutable snapshot-копии с label и ISO timestamp. Baseline-файлы по-прежнему создаются (backward compatibility), но snapshot — primary artifact для сравнения.

### #47. heredoc в интерактивном терминале ненадёжен для длинных файлов

**Симптом:** При создании markdown-артефактов через `cat > file << 'EOF'` содержимое обрывается.

**Причина:** Интерактивный терминал + длинные heredoc → проблемы с буфером.

**Решение:** Для длинных файлов использовать `printf`, Python one-liner, `nano`/`vim`, или scp готового файла. Не heredoc.

## Стратегический результат

1. **Least-privilege network perimeter** — Ollama больше не доступен всему LAN. Только конкретные клиенты (Windows + road-warrior). Enterprise-паттерн: whitelist вместо blacklist.

2. **Ops automation baseline** — nightly health-check + benchmark trending с immutable snapshots. При деградации — видно в логах до того, как пользователь заметит.

3. **Canary upgrade SOP** — формализованная процедура. Уже доказана на практике (0.20.0 → 0.20.2 без regression).

4. **Двухзонный доступ (LAN + WG)** — road-warrior может использовать стенд удалённо с тем же уровнем доступа, что и LAN-клиент.
