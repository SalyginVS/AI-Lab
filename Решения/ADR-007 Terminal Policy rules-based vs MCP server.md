# ADR-007: Terminal Policy — rules-based vs отдельный MCP server

## Контекст
Этап 8B планировался как «Terminal MCP». Обнаружилось, что Continue Agent уже умеет выполнять команды через встроенный VS Code terminal. Стоял выбор: разворачивать отдельный Terminal MCP server (например, `mcp-server-terminal`) или зафиксировать поведение через global rules.

## Решение
Использовать **rules-based terminal policy baseline** через `01-general.md` (`alwaysApply: true`). Отдельный MCP server не разворачивался.

Обоснование: встроенных возможностей Continue terminal достаточно для текущих задач. Добавление отдельного MCP server увеличивает сложность (ещё один STDIO-процесс, конфиг, потенциальные CVE) без прироста функциональности на данном этапе.

## Альтернативы

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **rules-based (выбрано)** | Нет новых зависимостей, мгновенная установка, легко откатить | Мягкая защита — модель может нарушить rule |
| Отдельный Terminal MCP server | Жёсткая политика, whitelist/denylist на уровне сервера | Дополнительный компонент, сложнее, риски CVE |

## Последствия
- Terminal MCP row в таблице архитектуры помечается как **Active** с формулировкой «Terminal Policy (rules-based)»
- Жёсткий denylist путей (`.env`, `secrets/`) — отложен до этапа 13 (`03-security.md`)
- Residual issue: иногда модель оборачивает всю команду в кавычки и печатает вместо выполнения — known issue, не блокирует
- При появлении реальной потребности в whitelist-контроле команд — пересмотреть в пользу MCP server (ADR пересмотр)
