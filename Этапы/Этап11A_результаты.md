# Этап 11A — Text-to-SQL PoC — Результаты

**Дата:** 2026-04-04
**Трек:** F (AI-as-Interface)
**Статус:** ✅ Завершён

## Задача

Проверить жизнеспособность паттерна NL→SQL (Natural Language → SQL) на локальном контуре: SQLite + gateway + локальные модели. Получить измеримый benchmark, определить границы качества, типичные ошибки и потенциал prompt engineering.

---

## Стек PoC

- **БД:** SQLite (`~/llm-gateway/ondo_demo.db`), 6 таблиц / 171 запись, схема ONDO (энергетика + гостиницы + производство)
- **Gateway:** v0.10.0, `/v1/chat/completions`, порт 8000
- **Скрипты:** `text2sql_poc.py` (strict evaluator) → `text2sql_poc_v2.py` (business-aware evaluator) → `text2sql_poc_v3.py` (+ prompt addendum)
- **Тестовый набор:** 20 вопросов, 4 уровня сложности (simple → edge cases), включая safety-тест (Q19: DML injection) и hallucination-тест (Q20: вопрос вне схемы)
- **Read-only enforcement:** SQLite `PRAGMA query_only = ON` + regex-валидация (двойная защита)

## Ход эксперимента (хронология)

1. **Strict baseline (v1):** первый прогон выявил жёсткий evaluator — SA 20–35% из-за формальных несовпадений (порядок колонок, alias, is_active filter).
2. **Business-aware evaluator (v2):** добавлен note-aware acceptance, compare_results стал семантическим. SA выросла до 65–75%. Evaluator-policy вопрос закрыт.
3. **Prompt experiment (v3):** prompt addendum дал +15–20 п.п. SA. Model-agnostic эффект: работает на всех 3 моделях.
4. **Booking patch (точечная правка):** попытка закрыть Q10 через усиление booking-rule дала regression (SA 90% → 85% для qwen3-coder). Диагноз: prompt bleed / over-generalization.
5. **Решение: freeze.** Дальнейший micro-tuning prompt вошёл в зону diminishing returns. Корректная остановка.

## Ключевые находки

### F1. qwen3-coder:30b — лучший Text-to-SQL executor

Неожиданно для расклада ADR-012 (где Gemma 4 — quality-first), qwen3-coder:30b показал лучший baseline (SA 75% vs 70% у gemma4:31b) и самую низкую latency (0.4s vs 4.0s). Для Text-to-SQL задач, где deterministic code generation важнее reasoning, qwen3-coder остаётся сильнейшим.

### F2. Prompt engineering даёт +15–20 п.п. SA

Эффект model-agnostic: работает на всех 3 моделях. Но есть потолок: после ~90% SA дальнейшие точечные правки вызывают regressions в других кейсах. Это подтверждает теорию, что для >90% SA нужен следующий архитектурный слой (semantic layer / RAG / few-shot retrieval), а не бесконечный prompt polishing.

### F3. Evaluator-policy — критически важный компонент

Strict evaluator (v1) занижал SA в 2–3 раза из-за формальных несовпадений. Business-aware evaluator (v2) — необходимое условие для корректного измерения. Ключевой урок: **evaluator и model/prompt layer нельзя оптимизировать одновременно.**

### F4. Safety и hallucination — 100% pass rate

Все 3 модели корректно отказали в DML-генерации (Q19) и вернули CANNOT_ANSWER на вопрос вне схемы (Q20). Read-only PRAGMA + regex — двойная защита работает.

### F5. Prompt bleed — реальный риск

Точечная правка одного rule (booking revenue) вызвала regression на других кейсах. Это enterprise-значимый вывод: prompt engineering для NL-to-SQL не масштабируется линейно. Для production нужен retrieval-based подход (schema-grounded few-shot examples вместо монолитного prompt).

### F6. Latency profile для enterprise

| Модель | Latency | Enterprise-применимость |
|--------|---------|------------------------|
| qwen3-coder:30b | 0.4s | Интерактивный режим OK |
| gemma4:26b | 1.5s | Приемлемо для аналитики |
| gemma4:31b | 4.0s | Только batch/report |
