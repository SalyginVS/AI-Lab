# Грабли #52: Ghost rule file перекрывает правильный файл в Continue UI

## Симптом
Continue UI показывал `rules/08-security.md` вместо ожидаемого `03-security.md`.

## Причина
В `%USERPROFILE%\.continue\rules\` присутствовал пустой файл `08-security.md` (0 байт). Continue отображал его вместо корректного `03-security.md`.

## Решение
Удалить пустой файл, перезапустить VS Code (полный restart, не только Reload Window).

## Урок
Перед созданием нового rule — проверить `ls` директории rules/ на наличие конфликтующих файлов. Пустые файлы в rules/ не игнорируются — они затеняют правильные.
```

# Грабли #53: Manual RAG reindex не видит LLM_GATEWAY_TOKENS

## Симптом
`python3 indexer.py` → `RuntimeError: LLM_GATEWAY_TOKENS env var is not set for rag-mcp`

## Причина
После перехода на mandatory auth (14B) indexer.py при ручном запуске не получает env-переменные, которые systemd подставляет через EnvironmentFile.

## Решение
Перед ручным запуском indexer.py:
```bash
set -a
source ~/llm-gateway/.env
set +a
python3 indexer.py
```

## Урок
Любой скрипт, зависящий от env из systemd EnvironmentFile, при ручном запуске требует явного `source .env`. Документировать в ONBOARDING.md.
```

# Грабли #54: MCP streamable-http требует полный session flow

## Симптом
Raw curl `POST /mcp` с телом `tools/call` → `406 Not Acceptable` или `Bad Request: Missing session ID`.

## Причина
Streamable-http MCP транспорт — session-oriented. Нельзя вызвать tool напрямую без установки сессии.

## Корректная последовательность
1. `POST /mcp` body=`{"jsonrpc":"2.0","id":1,"method":"initialize",...}` + header `Accept: application/json, text/event-stream`
2. Из ответа извлечь header `mcp-session-id`
3. `POST /mcp` body=`{"jsonrpc":"2.0","method":"notifications/initialized"}` + header `Mcp-Session-Id: <id>`
4. `POST /mcp` body=`{"jsonrpc":"2.0","id":2,"method":"tools/call","params":{...}}` + header `Mcp-Session-Id: <id>`

## Урок
Для тестирования MCP tools лучше использовать Continue Agent mode (он управляет session автоматически), а не raw curl. Если curl необходим — полный 4-шаговый flow.
```

# Грабли #55: CRLF в shell-скриптах с Windows

## Симптом
`bash setup-check.sh` → `$'\r': command not found`, `set: pipefail invalid option name`

## Причина
Файл перенесён с Windows (через scp из C:\Temp) в формате CRLF (Windows line endings).

## Решение
```bash
sed -i 's/\r$//' ~/llm-gateway/scripts/setup-check.sh
```

## Урок
Все .sh файлы, проходящие через Windows, нормализовать перед первым запуском. Добавить в deployment checklist.

# Грабли #57: Ollama override пропавшие параметры

## Симптом
- setup-check.sh: FAIL `NUM_PARALLEL != 1`
- setup-check.ps1: FAIL direct Ollama path не доступен с Windows
- `systemctl show ollama -p Environment`: OLLAMA_NUM_PARALLEL отсутствует

## Причина
В override.conf отсутствовали OLLAMA_NUM_PARALLEL=1 и OLLAMA_HOST=0.0.0.0:11434. Вероятно потеряны при upgrade Ollama 0.20.0→0.20.2 (14A), несмотря на SOP ([[Грабли033]]).

## Следствия
- Ollama слушал только 127.0.0.1 → FIM autocomplete с Windows не работал
- NUM_PARALLEL не был ограничен → потенциально нестабильное поведение при конкурентных запросах

## Решение
Override восстановлен (7 переменных). Ollama перезапущен. Подтверждено через `systemctl show ollama -p Environment`.

## Урок
SOP Ollama Upgrade необходимо дополнить: после upgrade проверять ВСЕ 7 переменных override, а не только наличие файла. setup-check.sh делает это автоматически — запускать после каждого upgrade.
```

# Грабли #58: git.yml vs git.yaml naming drift

## Симптом
setup-check.ps1: FAIL `mcpServers/git.yaml not found`

## Причина
Файл был создан как `git.yml` (3-буквенное расширение). Continue принимает оба варианта, но паспорт и check-скрипты ожидали `.yaml`.

## Решение
Переименован `git.yml` → `git.yaml`. Continue подхватил без проблем.

## Урок
Convention: все YAML-файлы в `.continue/` используют расширение `.yaml` (4 буквы). Зафиксировать в 01-general.md или ONBOARDING.md.
```
