## Задача

Формализация orchestrator паттерна из standalone скрипта (`orchestrator.py`, Этап 8C) в gateway API endpoint. Enterprise-transferable: единый API-контракт для pipeline execution.

Два новых endpoints:
- `POST /v1/orchestrate` — запуск named pipeline
- `GET /v1/orchestrate/pipelines` — список доступных pipeline

## Результат

### 16-1. Новый модуль `gateway/orchestrate.py` ✅

11-й модуль пакета gateway (~310 строк). Архитектурный паттерн — **self-call**: каждый шаг pipeline выполняется через `POST /v1/chat/completions` на `localhost:8000` с service token (`LLM_ORCHESTRATOR_TOKEN`). Это обеспечивает прохождение каждого шага через auth, structured logging и metrics автоматически — без дублирования кода chat.py.

- `load_pipelines()` — загрузка `pipelines.yaml` при старте gateway (module-level)

- `strip_think_blocks()` — перенесён из orchestrator.py

## Дефекты при интеграции (7 штук)

### Дефект 1. .env path в briefing

Briefing указывал `~/.env`, фактически файл в `~/llm-gateway/.env`. Deployment drift в инструкциях.

### Дефект 2. Shell-unsafe JSON в .env (Грабли #49)

`LLM_GATEWAY_TOKENS={"token":"user"}` без кавычек — bash `source` ломает JSON. Fix: обернуть в single quotes `'...'`.

### Дефект 3. Import mismatch: logger (Грабли #50)

`orchestrate.py` написан с `from .errors import GatewayError, logger`, но `errors.py` не экспортирует logger. Fix: использовать `from .logging_config import logger`.

### Дефект 4. env vars не экспортированы для дочернего процесса

`source .env` без `set -a` не экспортирует переменные в Python subprocess. Fix: `set -a && source .env && set +a`.

### Дефект 5. Metrics field name mismatch (Грабли #50)

`collector.record()` ожидает ключ `endpoint`, а orchestrate.py передавал `path`. Fix: `"path"` → `"endpoint"`.

### Дефект 6. Structured logging pattern mismatch (Грабли #50)

`logger.info(json.dumps(log_data))` — JSON попадает в поле `message`, а не как structured extra. Gateway-паттерн: `logger.info("text", extra=log_data)`. Fix: финальный `llm_orchestrate` event переведён на `extra={}`.

### Дефект 7. jq multiline в test harness (Грабли #51)

`jq -R '...' | tail -1` берёт только `}` из pretty-printed JSON. Fix: `jq -cR` (compact output).

## Грабли

### #49. Shell .env JSON quoting

**Симптом:** `source ~/.env` → `json.decoder.JSONDecodeError` при валидации `LLM_GATEWAY_TOKENS`.

**Причина:** `LLM_GATEWAY_TOKENS={"sk-lab-xxx":"vladimir",...}` без внешних кавычек. Bash интерпретирует фигурные скобки и внутренние кавычки.

**Решение:** Обернуть JSON-значение в single quotes: `LLM_GATEWAY_TOKENS='{"sk-lab-xxx":"vladimir",...}'`. Systemd EnvironmentFile парсит оба формата корректно — single quotes не ломают systemd.

**Статус:** Fixed. Все JSON env vars в `.env` должны быть в single quotes.

### #50. New module integration contract mismatch (тройной)

**Симптом:** Три ошибки при интеграции нового модуля orchestrate.py: (1) `ImportError: cannot import name 'logger' from 'gateway.errors'`; (2) metrics endpoint bucket = `"unknown"` вместо `/v1/orchestrate`; (3) structured log event не парсится jq (JSON внутри message вместо structured fields).

**Причина:** orchestrate.py был написан Claude на основе документации и памяти о кодовой базе, а не по живому коду. Три несовпадения с реальным internal API: (1) logger живёт в `logging_config.py`, не в `errors.py`; (2) `MetricsCollector.record()` ожидает ключ `endpoint`, а не `path`; (3) gateway JSONFormatter собирает structured fields из `extra={}`, а не из `json.dumps()` в message.

**Решение:** Три точечные правки в orchestrate.py при деплое.

**Урок:** При написании нового модуля для существующего пакета — всегда проверять imports и internal contracts по live коду (scp с сервера), а не по документации. Усиление Грабель #32: mismatch может быть не только в файлах, но и в API-контрактах между модулями.

### #51. jq -R multiline + tail -1

**Симптом:** Тесты 10/11 в test_stage16.sh падали с `jq: parse error: Unmatched '}'`.

**Причина:** `jq -R 'fromjson? | select(...)' | tail -1` выдаёт pretty-printed multiline JSON. `tail -1` берёт только закрывающую `}`, что невалидный JSON.

**Решение:** Использовать `jq -cR` (compact single-line output). `tail -1` корректно берёт последний полный JSON object.

**Статус:** Fixed в test_stage16.sh на месте.

## ADR-016: Synchronous orchestrate с self-call pattern

**Контекст:** Формализация pipeline execution в gateway endpoint. Варианты: (A) синхронный HTTP; (B) async + polling (job queue).

**Решение:** Вариант A — синхронный.

**Обоснование:** (1) `OLLAMA_NUM_PARALLEL=1` — Ollama физически выполняет один запрос, async не даёт параллелизма; (2) pipeline 60–90 сек, клиентский timeout 10 мин — запас достаточный; (3) async + polling добавляет job store, status endpoint, cleanup — overengineering для 2 пользователей.

**Self-call паттерн:** Каждый шаг pipeline → POST /v1/chat/completions на localhost:8000 с service token. Это обеспечивает auth, logging, metrics для каждого шага без дублирования кода. Trade-off: зависимость от gateway availability (если gateway падает mid-pipeline — partial failure). Mitigation: uvicorn async event loop обрабатывает self-call без deadlock.

**Последствия:** Третий user `orchestrator` в .env. Audit trail: caller user_id в `llm_orchestrate`, service user_id в `llm_completion`.

**Эволюция:** При масштабировании до 5+ пользователей или при добавлении `NUM_PARALLEL > 1` — пересмотреть в пользу async.

## Стратегический результат

**Orchestration формализован как API.** Standalone `orchestrator.py` (Этап 8C) остаётся для CLI-использования. Gateway `/v1/orchestrate` — для программного вызова из IDE, MCP, скриптов. Оба используют одни и те же `pipelines.yaml` определения.

**Self-call pattern = zero code duplication.** Pipeline steps проходят через тот же auth/logging/metrics контур, что и обычные chat-запросы. В audit trail видна полная цепочка: кто вызвал pipeline → какие модели отработали → сколько tokens потрачено.

**Enterprise-transferable.** API-контракт (POST pipeline+task → structured result) переносится на enterprise gateway без изменений. Единственная адаптация: pipeline definitions в yaml под enterprise модели.

**7 integration defects** обнаружены и устранены. Три из них (Грабли #49–51) — системные уроки для будущих этапов.
